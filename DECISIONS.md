# DECISIONS

Design notes and trade-offs for the `zama-indexer` confidential ERC-7984 indexer. See `TASK.md` (workspace root) for the brief this responds to.

## Overview

A confidential ERC-7984 indexer that watches one token on a local fhEVM stack, decrypts transfer amounts with the Zama SDK as events are indexed, and serves an ERC-20-style read API — cleartext balances, transfer history, and sync health — so a wallet partner consumes plain numbers and never touches FHE. Amounts the indexer is not yet entitled to decrypt are persisted encrypted and `pending`, never dropped, and backfilled automatically once an ACL delegation to the holder lands.

## Decision log

### 1. Indexer framework: Ponder (over a hand-rolled viem loop or a subgraph)

**Decision.** Use Ponder, a TypeScript-native indexing framework, as the spine. It hands us the load-bearing but unglamorous infrastructure for free — historical backfill, reorg-safe realtime sync, crash-resume from a checkpoint, an embedded database, and an extensible Hono server — which is exactly the brief's instruction to compose existing primitives rather than write an EVM indexer from scratch. Both hard requirements fit its event-driven model natively: confidential transfers and ACL delegation grants are both on-chain events (`ConfidentialTransfer`, `DelegatedForUserDecryption`, each with an SDK decoder), so inline best-effort decryption and event-triggered backfill are just handlers updating the same reorg-tracked tables, and a single process both indexes and serves the read API — the tightest "how cleanly you wire them together" story.

**Trade-offs.** Ponder owns the database, so we live inside its schema and store rather than bringing our own ORM, and side-effects in the indexing path are replayed on reorg — acceptable here because we decrypt via the SDK's local `cleartext()` transport (a cheap, deterministic read), but on Sepolia a relayer round-trip would couple sync speed to relayer latency; that coupling is our documented least-confident-under-load risk, and the production fix is a decoupled, throttled decrypt worker. It is also pre-1.0 (0.16, an unfrozen API), mitigated by pinning the version and anchoring to current docs over memory.

**Alternatives considered.** Each was weighed against Ponder and rejected for one decisive reason:

- **viem + cursor loop** — would work, but the cost is manual wiring: we would hand-build sync, reorg, resume, _and_ the HTTP server that Ponder gives out of the box. Control we don't need, paid for by rebuilding plumbing on a 3–4h budget.
- **The Graph** — disqualified outright (not a lateral move): sandboxed subgraph mappings cannot make the network calls to the Zama relayer that decryption requires, and its auto-generated GraphQL undercuts the API design the brief explicitly grades.
- **Envio (HyperIndex)** — the closest peer: same TypeScript handler model, its own very fast sync layer (HyperSync), automatic reorg handling. But it has no extensible app server out of the box — it serves an auto-generated GraphQL API, so the bespoke partner REST endpoints would force us to wire in a second HTTP library alongside it and carry a GraphQL layer we never use.
- **Subsquid** — heavier still: a separate batch-processor service plus a TypeORM model and a generated GraphQL API. Same gap as Envio (bespoke REST means bolting on another server), with more moving parts than a one-process app warrants.

Ponder is the only option that bundles the indexing infrastructure _and_ an extensible Hono server in one process — it fits the task simply, with no extra libraries and no unused GraphQL overhead.

### 2. Chain & test environment: local-only fhEVM stack (Anvil + forge-fhevm)

**Decision.** Target a local fhEVM stack exclusively (Anvil + forge-fhevm) with the SDK's `cleartext()` transport. The brief lets us pick whichever chain iterates fastest, and local wins on the axes that decide a graded POC: the chain is fresh and fully under our control, so tests deploy the contract and build a curated history (a shield, transfers, a delegation grant) deterministically and in advance; there is no faucet, so we fund EOAs and mint/transfer toy amounts inside the test setup itself; and there is no external dependency on a testnet RPC, faucet, relayer rate limits, or shared-chain timing. The decisive point is that `cleartext()` is not a mock: it drives the real high-level decrypt path and enforces real on-chain ACL (`persistAllowed` / `isAllowedForDecryption` / `isHandleDelegatedForUserDecryption`), so the pending → grant → backfill core — the heart of the task — is genuinely exercised, deterministically.

**Trade-offs.** Local does not exercise real FHE cryptography, ZK input proofs, or the hosted relayer/KMS round-trip, so we get no signal on relayer latency, throughput, 429s, or KMS availability — the under-load behaviour flagged as least-confident in Decision 1. We removed the Sepolia (`node()`) transport entirely rather than ship it behind a `MODE` flag we never ran: a dual-mode seam carried an unverified "one env var away" claim and, worse, an unavoidable ambiguity in error classification — the SDK's `DecryptionFailedError` means "not entitled" on `cleartext()` but "transient transport failure" on `node()`, so a single transport-agnostic `classify` cannot be correct for both. Going local-only makes the seam honest and unambiguous; the relayer-path realities (latency, 429s, reorg-replayed decrypts, KMS availability) and the decoupled decrypt-worker that would absorb them are documented as deferred production work (Decision 1), not pretended-at in code.

### 3. Read API is read-only and crafts no transactions — the delegation hint is declarative, not calldata

**Decision.** When the indexer lacks rights to decrypt an address's amounts, the API does not just flag the gap — it points the partner at the fix. The transfers feed sets `delegationRequired: true` (and balance returns `status: "indeterminate"`), and the OpenAPI docs for those fields describe, _declaratively_, the delegation that would grant access: delegate user-decryption on the token to the indexer's holder, naming the holder (`delegateAddress`) and token (`contractAddress`) and pointing at the SDK method — but never an unsigned transaction or signable calldata. The client constructs and signs the delegation itself via `@zama-fhe/sdk` (`delegations.delegateDecryption`). The API stays entirely on the read side of the trust boundary; the existing `DelegatedForUserDecryption` handler then backfills the cleartext automatically once the grant lands.

**Why (security).** Returning signable bytes from a server and having a client sign them is the blind-signing footgun behind drainer-class attacks: a compromised or MITM'd API could swap in an arbitrary malicious transaction, and a client that signs without fully parsing it is hacked. Keeping transaction construction client-side means the signing wallet only ever signs an operation it built and understands — always a decryption-delegation, never a transfer or approval — so a compromised API's blast radius collapses from "arbitrary transaction / fund loss" to, at worst, a _misdirected_ decryption delegation. Even that closes if the client verifies the delegate (the holder) against the indexer's holder identity known out-of-band (docs / pinned config), rather than trusting the response alone.

**Trade-off (UX), taken deliberately.** The point of a cleartext API is that the partner never has to understand FHE — and for _reads_ that holds completely. We could extend the same "never learn the underlying" UX to the write path by crafting the delegation transaction server-side. We deliberately do not: for a _signing_ operation the calculus inverts, and security dominates convenience. So this one write step requires the client to engage the SDK; everything else stays FHE-ignorant. The cost is a slightly heavier first integration at the grant step, in exchange for never placing the API in the signing path.

### 4. Balance is read from the live `confidentialBalanceOf` handle, not derived from the transfer log

**Decision.** The balance endpoint reads `confidentialBalanceOf(address)` and decrypts that one euint64 handle, rather than summing decrypted transfer deltas. The holder decrypts its own handle directly; for any other address it attempts the delegated path, and "not entitled" returns `indeterminate` (how to unlock — grant a delegation to the holder — is documented in the OpenAPI docs, see Decision 6, not echoed in the body). An uninitialised handle is treated as `0`.

**Why.** It is authoritative — no risk of missing or double-counting a balance-affecting event (transfers, wrap mints, unwrap burns) across two event streams — and it has no partial state: one value, one entitlement check, so the "decrypted some-but-not-all of an address's transfers" ambiguity never arises. The handle reveals only what the on-chain ACL already permits, and the endpoint is db-free (correct under index lag). Trade-off: current balance only (no balance-at-block), and a live RPC + decrypt per request — both acceptable here; a cache keyed off the public from/to logs is the documented next step (see "What I cut").

### 5. Transfer history is a pure reader; undecryptable rows are surfaced, never lazy-decrypted

**Decision.** `GET /v1/addresses/:address/transfers` serves stored cleartext + per-row `status` (`decrypted` | `pending` | `failed`) straight from the single `transfer` table — it never decrypts on read. Shields and unshields are the `mint`/`burn` rows of that same table (from/to == 0x0), so the feed is one ordered stream rather than a join of two (see Decision 7; the wrapper only mints via `wrap` and burns via `unwrap`, so `mint ⟺ shield` / `burn ⟺ unshield` is exact). Undecryptable rows return `amount: null` + a status, and a single-lookup `delegationRequired` boolean flags whether a delegation would unlock hidden amounts (true when a non-holder address has `pending` rows). Newest-first, keyset-cursor paginated over `(blockNumber, logIndex)`.

**Why pure-reader, not live-decrypt (as balance is).** Balance live-decrypts because it is one bounded handle; a history page is N handles, so decrypting on read would recouple reads to relayer latency/load, make response time non-deterministic, and duplicate the indexer's job. Decryption is the indexer's responsibility — inline best-effort, event-driven backfill on a delegation grant, and a bounded block sweep that retries transient `failed` rows on the direct path (the single-table simplification reverted the delegation-aware level-triggered retry — see Decision 7).

**Notify → trigger → background → observe.** Pending amounts are never dropped or faked. The client is notified (`status` + `delegationRequired`), triggers decryption out-of-band by granting a user-decryption delegation to the holder (client-side via the SDK), and the indexer backfills asynchronously; the client polls. No synchronous decrypt, and deliberately no `POST …/decrypt` endpoint — redundant when the holder is already entitled (the indexer auto-retries) and futile when it is not (a grant is required first), so it would only add a write-ish surface and an abuse vector.

**Consistency model.** Balance is the authoritative live number for one address at `asOfBlock`; history is an event log with per-amount decryption state, as-of `indexedBlock`. They are not a numeric identity: a client-side Σ (in − out + shields − unshields) equals the balance iff every matching row is `decrypted` (none `pending`/`failed`) ∧ history is caught up to the balance's block. Because an unshield is the burn row (emitted at unwrap-request, when the balance also drops), the feed and balance move together — no finalize-lag caveat in this model.

### 6. In-band code, out-of-band semantics — the in-band signal collapsed to a single `delegationRequired` flag

**Decision.** Responses carry machine-readable state and the runtime data a client acts on; the _meaning_ of each status / `delegationRequired` / error code and the human _how-to_ live in an OpenAPI 3.1 contract served at `/docs` (Swagger UI) and `/openapi.json`. Errors are RFC 9457 Problem Details (`application/problem+json`, a `type` URI into the docs plus a stable `code`). This is applied API-wide — balance, transfers, and health.

**Why.** Prose shipped in every response is an anti-pattern: static English bloats payloads, cannot be localised, and tempts partners into parsing it — after which it cannot change without breaking them. A stable code plus a docs pointer decouples the two — the Stripe / RFC 9457 pattern.

**We went further and collapsed it to one boolean.** Earlier iterations returned an `unlock` object (`action` + `delegateTo` + `contract` + `docs`), then a richer `reconciliation` summary, on the `indeterminate` / `pending` state. Once the OpenAPI docs existed both were redundant: `contract` is already the response's `token`, and the delegate is the indexer's holder identity — which a client must verify out-of-band against published / pinned config anyway, _not_ trust from the response. So the only in-band signal kept is the transfers feed's `delegationRequired` boolean (and balance's `status: "indeterminate"`); the OpenAPI `delegationRequired` docs publish this deployment's holder + token + the `delegations.delegateDecryption` how-to + the verify-the-holder security note. Less surface, identical actionability.

### 7. One activity table — simplicity over the public-amount overlay

**Decision.** Collapse to a single `transfer` ledger. Shields/unshields are not a separate stream: every `wrap` emits a `ConfidentialTransfer` mint (`from == 0x0`) and every `unwrap` a burn (`to == 0x0`) — verified in `ERC7984._update` — so that activity already lives in `transfer` as `mint`/`burn` rows. We therefore dropped the `shieldActivity` table (and the `Wrap` / `UnwrapFinalized` handlers feeding it), the persisted `delegation` table, and the `RevokedDelegationForUserDecryption` handler. The history feed is one ordered stream with a single `(blockNumber, logIndex)` cursor — no cross-table UNION/merge, no dual-cursor coordination, no dedup of mint/burn twins.

**What we gave up, deliberately.** (1) Shield/unshield amounts no longer arrive for free as public cleartext (`Wrap.roundedAmount` / `UnwrapFinalized.cleartextAmount`); they ride the normal decrypt lifecycle like any transfer — `decrypted` for the holder's own and delegated addresses (the partner case), `pending` for third parties the indexer is not entitled to. (2) the action signal degrades from a 3-valued entitlement (`self | delegated | none`) to a single `delegationRequired` boolean: without the `delegation` table we can't distinguish an already-delegated address from a never-entitled one (both non-holder addresses with `pending` rows simply read `delegationRequired: true`; their rows still backfill on the grant event — the brief's requirement holds — the API just can't pre-label which delegations are already active). (3) The level-triggered retry that healed already-active-delegation rows reverts to the simple `failed → direct` sweep; event-driven backfill-on-grant remains the entitlement path.

**Why it is worth it.** The single table is the leanest thing that satisfies the brief (track confidential transfers and shield/unshield activity; never drop an undecryptable amount; serve balance + history + health). For a wallet partner reading _its own_ users' history, the holder is entitled (directly or by delegation), so shield/unshield amounts decrypt anyway and nothing is lost in practice. The cost falls only on third-party amounts the API was never the authority for.

**Evolution.** To recover the public-amount benefit, re-index the public `Wrap` / `UnwrapFinalized` events into a small side table and join it back onto the `mint`/`burn` rows (by `txHash`, and `unwrapRequestId == burn handle` for unwraps) — surfacing cleartext shield/unshield amounts even where the indexer holds no decrypt rights. Re-introducing a `delegation` table likewise restores a 3-valued entitlement signal (`self | delegated | none`) in place of the boolean, and the level-triggered heal. Both are additive; the single-table core stays.

### 8. Health endpoint is lag-only — `GET /v1/health` composes Ponder's own probes, it does not re-report decryption

**Decision.** The brief's third read — _"how healthy and how far behind is the indexer?"_ — is answered by `GET /v1/health` returning just `{ chainId, indexedBlock, headBlock, blocksBehind }`, where `blocksBehind = max(0, headBlock − indexedBlock)`. Health is the indexer's _operational_ state (sync lag), not its decryption progress: per-amount decrypt state already lives in the transfer feed (per-row `status` + the `delegationRequired` flag), and is not duplicated here. We deliberately cut two fields an earlier sketch carried: `secondsBehind` (block timestamps are meaningless on the local Anvil chain, which only mines on demand — lag is pure block-number arithmetic) and a `status: healthy|syncing|behind` verdict (`blocksBehind` already conveys lag, and exposing the raw number lets a partner pick its own tolerance instead of trusting a hidden server-side cutoff).

**Why a self-call to `/status`, not a private-table read.** `indexedBlock` is Ponder's authoritative sync checkpoint. Ponder stores it as a 75-char fixed-width string in its internal `_ponder_checkpoint` table and exposes it, decoded, at its built-in `/status` route — but the decode helpers are unreachable (its `exports` map ships only the package root), so reading the table directly would mean re-deriving Ponder's namespace, column naming, and checkpoint encoding by hand and silently breaking on any bump. The handler instead self-calls `/status` (origin taken from the inbound request, so no port-guessing) — the same internal source through Ponder's stable accessor, the same pattern the test harness already uses. `headBlock` comes from `publicClients` (`ponder:api`), keeping health decoupled from the SDK/decrypt client entirely.

**Notify, don't verdict; 200 unless we genuinely can't answer.** The endpoint always returns `200` with the lag — _behind_ is a number a dashboard reads, not a failure; it returns `503` (`chain_unavailable`) only when the chain head itself can't be read, i.e. lag is unknowable. The one signal `blocksBehind` can't express — _still doing initial backfill_ vs. _was caught up and fell behind_ — is exactly Ponder's built-in `/ready` (200 once backfilled), so health defers to it rather than reproducing it. The result is three composed Ponder primitives — `/status`, `publicClients`, `/ready` — and one inline line of our own logic (the lag clamp), proven end-to-end against a booted indexer.

## What I composed vs. wrote myself

**Composed (off-the-shelf).** Ponder for the indexing spine — historical backfill, reorg-safe realtime sync, crash-resume, the embedded database, and the Hono server, all in one process (Decision 1); `@zama-fhe/sdk` for every decrypt and ACL operation, so the seam never re-implements FHE; Hono + `@hono/zod-openapi` for the REST surface and its generated OpenAPI contract; viem for chain reads; forge-fhevm for the local cleartext stack.

**Wrote by hand.** The project-specific glue no primitive provides: the decrypt seam (`ZamaClient` + the pure, unit-tested `classifyDecryptError`) that maps an SDK result onto a persistable `decrypted | pending | failed` status; the three Ponder handlers (inline decrypt, delegation-triggered backfill, bounded retry sweep) over a single `transfer` table (Decision 7); the read API — balance, transfers, health — with keyset pagination, the RFC 9457 error layer, and the `delegationRequired` / `indeterminate` signals that surface undecryptable state (Decisions 4–6, 8); and the self-provisioning integration-test harness. The heavy lifting stays in the libraries.

## Key trade-offs

**Storage.** Ponder owns the database, so the model lives in its store (Decision 1). One `transfer` table is the whole activity ledger — P2P transfers plus shield/unshield as mint/burn rows — with the encrypted handle always persisted and cleartext filled in place (Decision 7). Balances are not materialised: the endpoint reads the live `confidentialBalanceOf` handle and decrypts it, trading a per-request RPC + decrypt for an authoritative, partial-state-free number (Decision 4).

**Throttling.** On the local `cleartext()` transport a decrypt is a cheap, deterministic read, so the inline indexing path is not throttled. A Sepolia/relayer build would couple sync to relayer latency and 429s; the fix is a decoupled, rate-limited decrypt worker rather than inline decrypts in the reorg-replayed handler path (Decision 1). API-side rate limiting is a separate deferred concern ("What I cut" #6).

**Surfacing the in-between.** Undecryptable amounts are first-class — per-row `decrypted | pending | failed`, balance `indeterminate`, and a history `delegationRequired` flag as the one action signal — never a silent drop or a wrong number (Decisions 5–6; mechanism under "Un-decryptable events").

**Where I'd push back on the brief.** Two places. (1) The brief leans toward a testnet; for a graded POC a local cleartext stack is the better substrate — it exercises the real ACL/backfill core deterministically, where Sepolia adds faucet/relayer flakiness without new signal (Decision 2). (2) "The partner never learns FHE" holds for reads but should not extend to the one write step, granting decrypt rights: crafting that transaction server-side would put the API in the signing path, so the partner engages the SDK for that single grant by design (Decision 3).

## Un-decryptable events & backfill

The core domain rule, in one place. On every `ConfidentialTransfer` the handler stores a row with the encrypted `amountHandle` and attempts a direct decrypt; if the holder is not entitled, the row is persisted `pending` (or `failed` for a transient error) with `amountClear` null — the handle is never discarded, so no amount is lost. Two triggers heal pending rows: the `DelegatedForUserDecryption` handler (scoped by config to grants made _to the holder_) backfills the delegator's pending rows via the delegated path the moment a grant lands; a bounded block sweep retries `failed` rows on the direct path for transient hiccups. The insert is `onConflictDoNothing` and decryption is idempotent, so replay and backfill never double-write. The sharp edges in this path — a classification false-positive stranding a row, and the `MAX_ATTEMPTS` cap — are catalogued under "Least confident" and "What I cut" (#1–3).

## Reflection

### Least confident under partner load

**The inline decrypt on the `ConfidentialTransfer` handler (`src/index.ts`).** Every transfer is decrypted synchronously inside the indexing handler, before the row is written (`await zama.decryptTransfer(handle, TOKEN, [from, to])`). Ponder runs handlers serially per chain and replays them on reorg, so that `await` sits on the indexer's critical path: sync throughput is capped at roughly one decrypt round-trip per event. On the local `cleartext()` transport that round-trip is a cheap, deterministic read, so it never bites — which is precisely why I trust it least: its load behaviour is unobserved by construction, and the production (`node()`/relayer) build turns the same call into a network hop to the relayer/KMS. This is the coupling flagged in Decisions 1–2, named here as the single piece I'd watch first under load.

Two properties make it worse than one round-trip per event. **First, the work is `O(all transfers on the token)`, not `O(transfers the holder cares about)`:** `decryptTransfer` (`src/utils/zama.ts`) tries the direct path and, on the `pending` miss, probes each party's on-chain `isActive` delegation before any delegated decrypt — so a transfer between two accounts the indexer can't decrypt still pays a direct decrypt attempt plus an `isActive` read per party (~3 round-trips) only to land `pending`. A busy token whose traffic is mostly between non-delegating third parties pays full freight per event for nothing. **Second, reorg replay re-runs those decrypts,** burning relayer quota and re-tripping rate limits on blocks already paid for.

**What breaks first.** Under a transfer burst — or simply after the relayer migration — per-event handler time exceeds the inter-block budget, indexing throughput falls below the token's event rate, and `blocksBehind` grows without bound. The transfer history, served from the index, is what degrades: recent transfers arrive late and sit `pending` longer, so `GET /transfers` silently returns an incomplete, ever-staler feed. (Balance is a live on-chain read, correct under lag by design — Decision 4 — so it stays fresh; its own cost is the separate per-request decrypt.) The product's one job — fresh cleartext out — degrades first on the history path, and quietly: `/v1/health` _reports_ the lag, but nothing in the pipeline sheds the load that causes it.

**How I'd prove it.** Wrap `decrypt` with an injected delay to simulate relayer RTT (or point the seam at `node()`), replay a block range carrying K transfers per block, and watch `/v1/health.blocksBehind` climb and fail to recover once arrival rate exceeds `1/RTT`. The clean metric is indexing throughput (events/sec, instrumented on the handler) against the token's peak tx/sec — the moment the former drops below the latter, lag is unbounded. The durable fix is the one named in Decisions 1–2: move decryption off the handler into a decoupled, rate-limited worker (the handler persists the row and enqueues the handle), so sync speed is independent of decrypt latency, replay stays cheap and idempotent, and the worker — not the chain's event rate — sets the relayer call rate.

**A correctness risk the same load amplifies: the not-entitled branch of `classifyDecryptError`.** ACL denial is the one decrypt outcome the SDK gives no code for — on `cleartext()` it surfaces as a generic `DecryptionFailedError` separable only by message (`not authorized` / `not delegated`), so the pending-vs-failed call rests on a regex coupled to SDK-internal strings (`#assertDecryptAuthorization` / `#assertDelegation`). The hazard is silent and asymmetric: a not-entitled handle misread as `failed` is retried only on the direct path, capped at `MAX_ATTEMPTS`, then stranded and invisible to grant-backfill (which scans `pending`) — permanent cleartext loss; a transient error misread as `pending` waits on a grant that never comes. A relayer build's 429s, KMS latency, and propagation races are exactly the ambiguous errors this branch never sees on the local stack, so load is what would expose it. How I'd prove it: fault-inject the SDK's real throws (the "stranger → pending" integration test is the seed) and assert every error maps to a _recoverable_ status; the durable fix keys retry eligibility off `amountClear IS NULL` rather than the status label, so a misclassification can no longer strand data ("What I cut" #1–2).

### What I cut / the next four hours

_What was deliberately left out, and what I would do first with more time._

The seam that got the most scrutiny this round was decrypt error handling, but its correctness _verification_ was deliberately time-boxed. What shipped: the decrypt classifier — extracted as the pure, unit-tested `classifyDecryptError` — now routes on the SDK's machine-readable `ZamaError.code` via `matchZamaError` instead of `instanceof` + message cascades — fatal misconfiguration codes (`CHAIN_MISMATCH`, `SIGNER_NOT_CONFIGURED`, `CONFIGURATION`) rethrow so a deploy-time mistake fails loud; the unknown-error default flipped from a silent `pending` to `failed` (so a misclassified row is sweep-retried and honestly labelled, not parked looking like "awaiting authorization"); and the relayer-only codes (`DELEGATION_NOT_PROPAGATED`, `RELAYER_REQUEST_FAILED`, `NO_CIPHERTEXT`) are mapped forward-looking. What was delayed, in priority order:

1. **Harden the not-entitled classification — the _correctness_ I trust least (the secondary risk under "Least confident").** What shipped: `classifyDecryptError` extracted as a pure function and pinned by a fast unit suite (`test/unit/classify.test.ts`, `pnpm test:unit`) — the not-entitled message regex pinned by a golden string (the `not delegated` branch in the unit suite; the companion `not authorized` branch is canaried live in the integration test), mutation-tested to confirm it catches a reworded regex, a removed fatal-throw, and any regression to the silent-`pending` default; the "stranger → pending" integration test is the live canary against an SDK reword. Deferred: the relayer-path cases — 429/5xx, KMS latency, the delegation-propagation race — that the local `cleartext()` stack cannot fire, so they stay entirely unexercised.
2. **Make retry eligibility key off the data invariant, not the status label — the change that makes (1) stop mattering.** The stranding in (1) exists only because the label gates which retry path a row gets: `RetryDecryptions:block` retries `status = 'failed'` rows via the direct path (no delegator), and the `DelegatedForUserDecryption` backfill retries `status = 'pending'` rows only — so a misclassification decides whether a row is _ever_ retrievable. The fix is to drive both triggers off `amountClear IS NULL`: the sweep consults the `delegation` table and retries via the _delegated_ path when a row's `from`/`to` has an active delegation, and the backfill scans `amountClear IS NULL` touching the delegator rather than `status = 'pending'`. Then classification degrades to an advisory DX/throttling hint that can be wrong without losing data.
3. **Never silently give up at `MAX_ATTEMPTS`.** A `failed` row that exhausts its five sweep retries drops out of the sweep query forever — indistinguishable, from the API's side, from "still trying." Either retry forever with a decaying cadence, or promote exhausted rows to an explicit terminal state surfaced on the health endpoint for operator re-trigger; "capped and forgotten" violates the brief's never-silently-drop rule.
4. **Now built (was the biggest cut).** The partner-facing read API — cleartext `balance`, a unified `transfers` history, and a bespoke `/v1/health` (how-far-behind/lag beyond Ponder's built-in `/status`), plus the brief's end-to-end "event in → cleartext out of the API" happy-path e2e — is now implemented and green.
5. **Balance caching, with provably-correct invalidation — the balance endpoint's first optimisation under load.** Once the live-read balance endpoint lands, the obvious next step is caching the last decrypted balance per address — and the confidential setting makes the invalidation _provably_ correct, not heuristic. A confidential balance moves _only_ via a `ConfidentialTransfer` (including mint/burn), and that event's `from`/`to` are **public** indexed topics (only the amount is encrypted), so the indexer already records, in cleartext, every block at which an address's balance could have changed. Cache `{ balance, asOfBlock }`; on read, if no indexed transfer touches the address after `asOfBlock` the balance is unchanged and the cache is served at zero RPC/decrypt cost, otherwise re-read and re-decrypt. The one freshness caveat is indexer lag — the oracle is only as current as the indexer's sync height — so cache hits gate on the indexer being caught up to head (the lag the health endpoint reports). Deferred because live-per-request is correct and simplest here; it collapses the hot path (repeat balance reads) to a single public-log lookup.
6. **No auth, rate-limiting, or CORS on the partner-facing surface — out of scope for a local POC, but the first hardening a real deployment needs.** The read API ships with no API-key/JWT authentication, no per-caller rate limiting, and no CORS policy, and nothing in the current docs flags their absence. For a local, single-tenant POC against an Anvil chain this is defensible — there is no untrusted caller, no quota to enforce, no browser origin to gate, and the brief explicitly scopes ops out ("We are not testing your ops setup") — so the submission neither needs nor claims them. A genuinely partner-facing surface needs all three: authentication to bind each call to a partner identity, rate limiting to bound the live RPC-plus-decrypt cost the balance endpoint pays per request (and to throttle the relayer round-trip a Sepolia build would add — Decision 1's least-confident-under-load risk), and a CORS policy if any browser client calls it directly. A production pass would add them and document the trust boundary they draw.

### SDK feedback

_Two or three concrete improvements to `@zama-fhe/sdk`. For each: (a) the concrete change, (b) the partner-integration scenario it unblocks, (c) its priority relative to the others._

Credit before the asks (this is a design-review note, not a grievance list): the typed-error layer is a genuinely good addition. Every SDK error extends `ZamaError` with a stable, machine-readable `.code` (the `ZamaErrorCode` enum) and `matchZamaError` ships as a code-keyed dispatcher (verified against the installed `3.1.0-alpha.15`; source in `errors/base.ts`). The payoff is concrete: after I routed the decrypt seam through `.code`, every failure the SDK assigns a code to is handled by branching on that code with **zero string matching** — exactly the contract a backend integrator wants, and a real step up from sniffing `.message`. That strength is what makes the one exception worth flagging in #2: the single decrypt outcome with _no_ code — ACL "not entitled" — is the one this indexer leans on most, so it stays the lone place we're forced back to matching message strings.

1. **Export the contract ABIs (or add a `@zama-fhe/sdk/abi` entry).** The package ships event decoders (`decodeConfidentialTransfer`…), topic constants (`Topics`, `ACL_TOPICS`), and pure call-builders (`confidentialTransferContract`…) — 123 barrel exports — but **no consumable ABI array** for the ERC-7984 wrapper or the ACL contract (verified against the installed `3.1.0-alpha.15`: zero ABI-named exports, nothing under `dist/*abi*`, no `./abi` subpath). **(a) Change:** export `confidentialWrapperAbi` / `aclAbi` from the barrel or a `/abi` subpath — they already exist in-repo at `packages/sdk/src/abi/*.abi.ts` — and include the ACL delegation _events_, since the shipped `acl` ABI is functions-only. **(b) Unblocks:** wiring any ABI-driven indexer or tool — Ponder, a subgraph, viem `getContract`/`watchContract`, wagmi — against an ERC-7984 token; today you vendor the ABI out of the repo or hand-author event fragments (I had to hand-write the `DelegatedForUserDecryption` event ABI just to index delegations). That is step one of the exact "wallet partner builds an indexer" use case this SDK targets. **(c) Priority:** **high** — first blocker for backend/indexer integration, and near-free to fix since the ABIs already live in the repo.
2. **Give decrypt errors a retryability signal and a distinct "authorized-decryption denied" type.** The codes credited above tell you _what_ failed; the gap is one rung up — deciding _what to do_ about it — which is exactly the decision this indexer makes on every undecryptable transfer. Two signals are missing, and both bite here. **(a) Change:** (i) expose the SDK's own transient detection — `isTransientError` (`relayer/relayer-utils.ts`) is internal today and is the SDK's _only_ message-regex; lifting it to a `retryable: boolean` on `ZamaError` (or an exported guard) lets callers reuse it instead of re-deriving it; (ii) split "no ACL / authorized-decryption denied" out of the generic 4xx `RelayerRequestFailedError` into its own code, and stop collapsing status-less transport failures into `DecryptionFailedError` whose `.message` is a generic fallback (the real cause is reachable only via `.cause`). **(b) Unblocks:** the exact pending-vs-failed decision this indexer makes on every undecryptable transfer — "park and wait for an ACL grant" vs "transient, retry on the next block sweep." Without the signal an integrator hand-rolls `classify(err)` and gets it subtly wrong: on the `cleartext()` transport the not-entitled `DecryptionFailedError` is separable from a genuine transport failure only by its message text, so the pending-vs-failed call hinges on a brittle `err.message` regex (this project's `src/utils/zama.ts` does exactly that — and it breaks the day the SDK rewords a string). **(c) Priority:** **medium** — the typed errors already make a correct classifier _possible_; this makes it _easy and uniform_ and removes the one place every backend integrator otherwise reinvents message-regex.
3. **Surface the local cleartext stack — clone-and-run is great DX, but you have to find it yourself.** forge-fhevm's local path is excellent: clone the repo and run `./deploy-local.sh` and the entire cleartext FHE host stack (Executor / ACL / InputVerifier / KMSVerifier) materialises on Anvil at canonical addresses with zero config; the SDK's own `contracts/` then adds a ready `Deploy.s.sol` that deploys the standard test tokens on top — together a turnkey, deterministic local test chain that needs no testnet, faucet, or relayer. **(a) Change:** surface this from the high-level `@zama-fhe/sdk` itself — a "local testing" quickstart (clone forge-fhevm → `deploy-local.sh` → point `cleartext()` at it), and ideally a thin SDK helper/CLI that wraps the clone+deploy so integrators don't shell out to a sibling repo. **(b) Unblocks:** the first hour of any backend/indexer integration — today you only discover `cleartext()`, forge-fhevm, and the canonical host addresses by reading SDK source and the `node-viem` example (which is Sepolia/`node()`-only), so the local, relayer-free path that actually makes this project testable is the least documented one. **(c) Priority:** **high for backend DX** — the difference between "works in an afternoon" and "reverse-engineer the test setup," and it compounds with feedback #1 (export the ABIs) to make the whole indexer story copy-pasteable. One doc-line caveat: `deploy-local.sh` deploys the FHE _host_ stack only — your application token is still yours to deploy (the SDK's `contracts/Deploy.s.sol` is the perfect example to copy, which is exactly what this submission does).

### AI assistance

I built this with Claude Code (Claude Opus) as the primary tool, under one organising constraint: the alpha `@zama-fhe/sdk` postdates the model's training data, so the workflow is deliberately built to make it reason from a colocated source of truth instead of memory.

**Process.**

1. **Investigation before code.** The first pass was pure planning — comparing indexing primitives (Ponder over a hand-rolled viem loop or a subgraph), working out how the SDK actually performs user-decryption and ACL delegation, and weighing Sepolia against a local fhEVM stack. The output was the design doc and the Decision log above, not code.
2. **Colocated SDK source of truth.** Cloned `zama-ai/sdk` as a sibling of the project (`../sdk`) on the **`prerelease`** branch — the one that tracks the alpha — so the model reads real source and examples from disk (the `node-viem` example, the exported event decoders, `cleartext.ts`) rather than fetching stable docs that lag the alpha or inventing the API.
3. **Grounded project init.** Stood up a project `CLAUDE.md`, task-scoped skills (`ponder-indexer`, `zama-sdk`), and hooks, seeded from the brief (`TASK.md`, from the take-home email), the `sdk/` clone, and the Ponder scaffold — so every session opens with the domain rules and the never-edit-generated-files / never-generate-SDK-from-memory guardrails already loaded.
4. **PRD via superpowers.** Drove the model through the superpowers brainstorming → writing-plans pipeline to produce an in-depth design/PRD before touching implementation, instead of free-coding from a one-line prompt.
5. **Test-driven, mutation-validated implementation.** Implementation ran through the `superpowers:test-driven-development` skill — write the test, watch it fail, then the minimal code to green (the classifier's first unit test failed on a missing export before the function was extracted). Because tests authored alongside their code — by a model or a human — tend to pass vacuously, the model then ran an **AI-driven mutation loop** to validate them: for each critical behaviour, deliberately mutate the implementation (make a fatal error code stop throwing, drop a branch from the not-entitled regex, revert the unknown-error default to a silent `pending`), re-run the suite, confirm that mutation turns a _specific_ test red, then revert — looping until every guard is proven to bite. That proves the tests constrain behaviour rather than merely cover lines. The decrypt error classifier was extracted to a pure function (`classifyDecryptError`) and pinned by a fast, chain-free unit suite (`test/unit/classify.test.ts`, `pnpm test:unit`) built this way.

**Where the model was subtly wrong (and how I caught it).** The brief asks for one such place; two are worth recording, because both were the same _shape_ of mistake — plausible, idiomatic code that passed a happy-path test and was wrong only in the confidential-domain edge case that separates this from a transparent-token indexer.

1. **Deriving balance from the transfer log instead of reading the on-chain handle.** The model's first balance design reconstructed each address's balance by folding its history — Σ(in) − Σ(out), shields/unshields as mint/burn deltas — the idiomatic ERC-20-indexer pattern, and correct _whenever every contributing row is decrypted_, which is exactly why a naive test passes. It is silently wrong the instant any amount is `pending` or `failed`: the sum omits that delta and returns a confident wrong number — the precise failure the brief warns against, and the one a wallet partner would trust blindly. I asked it to instead read the SDK's authoritative `confidentialBalanceOf(address)` handle and decrypt that single euint64 (`ZamaClient.readBalance`): one value, one entitlement check, no partial-state ambiguity and no materialised-balance bookkeeping to keep in sync. That is now argued as settled design in **Decision 4** — this note records that the model had to be steered there rather than arriving at it.
2. **A decrypt seam that only used the holder-direct path.** The model's first decrypt implementation called only `decryption.decryptValues` — it decrypted amounts the holder was itself a _party_ to and nothing else, omitting the delegated path entirely. It _looked_ complete because the separate `DelegatedForUserDecryption` handler backfills on a new grant, which masks the gap: an amount the holder was _already_ entitled to via an existing ACL delegation would sit `pending` indefinitely — until some redundant later grant happened to re-fire the backfill — even though the chain already authorised it. I asked it to add the delegator case explicitly: at index time, probe the chain (`delegations.isActive({ contractAddress, delegatorAddress, delegateAddress })`) for each counterparty and, when a delegation is active, decrypt via the delegated path (`ZamaClient.decryptTransfer`), so an already-active delegation decrypts on the spot instead of never. The residual cost is one `isActive` call per otherwise-undecryptable transfer; the noted next step is to track delegations in a local table and read entitlement from there (Decision 7's _Evolution_), trading the per-transfer chain hit for indexed state.
